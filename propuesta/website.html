Nuevo sitio web (alineado al sistema POS)

El sitio actual no muestra el componente más importante: productos y menú. Además, el contenido clave está fragmentado, lo que afecta SEO, conversión y consistencia. Esta propuesta reordena el sitio para que sea un front operativo: menú + sucursales + pedido por canal + base para crecimiento multi-estado.

Objetivo
Convertir visitas en pedidos y datos (por sucursal/canal)
Prioridad
Tabasco (piloto) + expansión Veracruz/CDMX
Problemas detectados
  • Home incompleta: solo lista ciudades, sin direcciones ni menú.
  • Menú ausente: baja conversión y mal posicionamiento.
  • SEO técnico débil: sin sitemap/robots; contenido clave cargado por fragmentos.
  • CTAs confusos: “Pedido/Factura” apuntan al mismo lugar.
Solución: sitio modular, contenido renderizado completo, páginas por sucursal y menú indexable.

Estructura propuesta (sitemap)

Diseñado para crecer por estados/sucursales y conectar con el sistema POS.

1) Home
Conversión
  • Hero con propuesta clara y CTA “Pedir ahora”.
  • Atajos por ciudad/estado (Tabasco, Veracruz, CDMX).
  • Productos destacados (Top sellers).
Resultado: visitas → pedido en 1–2 clics.
2) Menú
SEO
  • Categorías: clásicos, especiales, dulces, bebidas.
  • Precios referenciales y disclaimers por sucursal/canal.
  • Etiquetas: “Clásico”, “Premium”, “Picante”, “Dulce”.
Resultado: posicionamiento por producto + claridad de oferta.
3) Sucursales
Operación
  • Directorio con filtros por estado/ciudad.
  • Página por sucursal: dirección, horarios, teléfono, links de pedido.
  • Mapa y botones por canal (Uber/Didi/Rappi/WhatsApp).
Resultado: pedido correcto a la sucursal correcta.

Dato clave

Operación por sede

  • Cada sucursal/franquicia puede tener su propia producción e inventario.
  • El sitio debe reflejar disponibilidad y enlaces por canal por sucursal.
Esto evita pedidos mal asignados y mejora la experiencia del cliente.

Conexión al sistema (sitio → POS → operación)

El sitio nuevo se diseña para integrarse naturalmente con el modelo POS propuesto.

Cliente

Un solo flujo de pedido

  • El usuario elige sucursal (por ubicación o estado).
  • Elige canal: mostrador, WhatsApp, Uber/Didi/Rappi (según disponibilidad).
  • El sitio manda el pedido al punto correcto (enlace directo o formulario).
Menos fricción = más conversión.
Operación

Datos por sucursal y canal

  • IDs de sucursal en el sitio para asociar pedidos.
  • UTMs por canal para medir performance por plataforma.
  • Base lista para tracking y facturación por canal/sucursal.
El sitio deja de ser “folleto” y se vuelve parte del sistema.
